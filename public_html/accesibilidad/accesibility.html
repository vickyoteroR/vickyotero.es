HERRAMIENTAS DE ACCESIBILIDAD

DENTRO DEL MUNDO WEB

  • Estructura semántica

    Es muy importante una estructura semática correcta para el lector de pantalla.

  • Marcado de idioma

    Con el fin de que el lector de pantalla lea correctamente el DOM, no se nos puede olvidar especificar el idioma correcto al principio

  • Imágenes

    • Todas las imágenes deben llevar el atributo alt =" " donde se describirá la imagen

    • igualmente se recomienda que el nombre de la imagen sea oritentativo. No poner nunca nombres del tipo alt="img-1"

    • Cuando las imágenes que son decorativas, es mejor dejar el alt=" " vacío, ya que que no aportaría nada describir una línea

  • aria-hidden / sr-only

    Una técnica importante para mejorar la experiencia de los usarios que necesitan tecnología asistencial, es garantizar que solo las partes que sean necesarias para un correcto entendimiento, se muestren a dicha tecnología.

    • De esta forma todo lo que tenga estilos visibility:hidden, display:none o utiliza el atributo de HTML 5 hidden se ocultará para los usuarios de tecnología asistencial mediante el atributo aria-hidden="true"

    • Sin embargo cuando queremos incluir texto solo para el lector de pantalla, pero no que esté visible en nuestra página, la ténica es utilizar class="sr-only"

  • aria-label / aria-labelledby / aria-describedby

    Con los lectores de pantalla, se pierde una gran cantidad de la información visual que se presenta en una página web. Debido a esto, tenemos que proporcionar información específica a lo que nos puede parecer obvio.

    Para ello debemos utilizar los atributos ARIA que son específicos para tal propósito

    • ARIA-LABEL: se utiliza en los casos en los casos en que no hay una etiqueta de texto visible en pantalla.

    • ARIA-LABELLEDBY: especifica el nombre o etiqueta del elemento actual.

    • ARIA-DESCRIBEDBY: en este caso la información que proporciona puede ser más descriptiva. A diferencia aria-labelledby, que espera un nombre para el elemento, aria-describedby permite toda una frase o párrafo.

  • Menú

    Siempre que creemos un menú o cualquier interacción, se debe implementar tanto teclado como para ratón.

  • Ventanas modales

    Las ventanas modales siempre se posicionan al final del DOM. Por dicho motivo, el lector no puede seguir interactuando, al no poder encontrar el contenido que debería obtener con la última acción realizada. Para evitar que el usuario tenga que recorrer todo el DOM hasta el final, estas son las medidas a tomar:

    • Quitar la página de detrás con aria ="hidden"

    • Evitar que con el teclado (TAB) nos saque de la ventana modal

  • Formularios

    Es muy importante que en los formularios se especifique el atrituto for="name" del label en concordancia con el id="name" del input .

  • CHECKBOX / RADIO BUTTON

    Cuando en un formulario tenemos varios checkbox / radio button, se deben agrupar en un fieldset con un lengend que determinará de que se tratan las opciones ofertadas.
